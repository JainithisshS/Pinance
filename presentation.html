Background
Problem Statement

User Pain Points

  • Bank SMS and UPI notifications arrive daily — spending tracking is entirely manual
  • Users cannot quantify personal financial risk (EMI load, savings rate, variable spend)
  • Financial news is presented without any link to the user's actual money situation
  • No accessible tool combines transaction data, live news, and domain knowledge into a single grounded recommendation

Why Existing Solutions Fall Short

  • Expense trackers — manual entry only, no intelligence or forward-looking insight
  • Market apps — generic news with no personalisation to individual finances
  • Financial advisors — inaccessible cost for students and early-career earners
  • No product bridges personal transaction data, real-time sentiment, and financial knowledge into one answer

Core Gap: No single system unifies personal transaction data + real-time news sentiment + financial domain knowledge into agentic, explainable, personalised guidance.

Agent A
Personal Finance Analyzer — SMS Parsing & Risk Model

SMS Parsing — 3-Tier Extraction

  • Tier 1: AMOUNT_PATTERN regex — currency prefix (Rs / INR / ₹) and keyword-prefixed amounts
  • Tier 2: keyword-proximity search within 15 chars of "debited" / "credited"
  • Tier 3: last-resort — any standalone number with 2+ digits
  • Credit / Debit: 15 credit keywords vs 12 debit keywords for correct income / expense split
  • Category: keyword map + TF-IDF + LR at ≥ 0.50 confidence threshold → 10 classes

Risk Model — Two-Tier Pipeline

  • Tier 1 ML: Random Forest (120 trees, depth 6) on 6 features — income, fixed_exp, variable_exp, savings, savings_rate, variable_share; trained on 900 synthetic profiles
  • Month proration: partial-month data projected to full-month to prevent early-month low-risk bias
  • Confidence bands: high ≥ 0.75 · moderate ≥ 0.50 · uncertain below
  • Tier 2 LLM: Groq generates 4–5 sentence, data-specific insight referencing actual user numbers
# Tier 1 — primary regex AMOUNT_PATTERN = re.compile( r"(?:inr|rs\.?|₹)\s*([0-9,]+\.?[0-9]*)|" r"([0-9,]+\.?[0-9]*)\s*rupees|" r"(?:debited|credited|spent)" r"(?:\s*(?:by|for|of|:)?\s*)" r"([0-9,]+\.?[0-9]*)", re.IGNORECASE, ) # Tier 2 — keyword proximity kb = re.search( r"(?:debited|credited)\D{0,15}" r"([0-9,]+\.?[0-9]*)", msg, re.I ) # Risk model feature vector MonthlyFeatures( income = 50_000, fixed_expenses = 18_000, # rent, EMI variable_exp = 12_000, # food, shopping savings = 20_000, savings_rate = 0.40, variable_share = 0.24, ) → predict_risk() → "low" (confidence=0.87, band="high")
Agent B
News and Market Intelligence

NLP Processing Pipeline

  • FinBERT (ProsusAI/finbert): tokenize → transformer encoder → softmax; headline weight 40% + article body weight 60%
  • spaCy NER: ORG entity extraction to identify company and sector from article text
  • 27 positive + 31 negative domain keyword fallback with strong-phrase detection
  • 25+ company-to-sector mappings: Banking, IT, Auto, Telecom, FMCG, Metals, Pharma
  • Move extraction: "up 3%", "fell 150 pts" via percentage and point regex patterns
  • 13 boilerplate-removal patterns strip editorial noise before FinBERT inference
  • Trend Model: Random Forest (120 trees, depth 5) on FinBERT aggregate features

Data Sources

  • Live RSS: Economic Times Markets feed — real-time financial headlines
  • Fallback corpus: 10 curated demo articles (Reliance, HDFC, TCS ...)
  • Live NSE data: pre-warmed at server startup, auto-refreshed on request
  • LLM summaries: Groq generates per-article market narrative with sentiment context

Output

  • Per-article: sentiment label + score + entities + move percentage
  • Aggregate: topic sentiment ratio → trend prediction (bullish / sideways / bearish)
  • NSE indices with sector-level breakdown

Agent C
Decision Synthesizer and RAG Chatbot

Tier 1 — RAG Knowledge Retrieval

  • 32-chunk curated corpus: emergency funds, 50/30/20 rule, debt avalanche / snowball, SIP investing, 80C / 80D tax saving, insurance, market crashes, scam detection, retirement planning
  • Custom TF-IDF with IDF smoothing + tag-match bonus (×3) for topical relevance
  • Top-3 most relevant chunks injected as grounding context in every LLM call

Tier 2 — Agentic LLM with Fallback

  • Multi-turn session via Groq LLaMA-3.3-70B-Versatile with full chat history
  • Context: Agent A financial data + Agent B sentiment + RAG chunks + history
  • System prompt enforces rupee amounts and references to real user numbers
  • Keyword-matched deterministic fallback covers 6 core topics when LLM unavailable

Learning Engine
Adaptive Micro-Learning System

Bayesian Knowledge Tracing (BKT)

  • 3-state belief: P(Unknown) + P(Partial) + P(Mastered) = 1.0 per concept
  • Fast correct (< 30s) → P(Mastered) += 0.30
  • Slow correct (> 30s) → P(Mastered) += 0.15, P(Partial) += 0.15
  • Fast wrong → P(Unknown) += 0.20, P(Partial) += 0.10 (guessing interpretation)
  • Slow wrong (> 60s) → P(Unknown) += 0.40 (struggling interpretation)
  • Belief renormalised after every update; persisted across sessions in DB

Curriculum Compiler Scoring

  • score = readiness × urgency × relevance
  • Readiness: minimum prerequisite mastery (threshold 0.60 to unlock)
  • Urgency: 1 − P(Mastered) — weakest concepts surfaced first
  • Relevance: high-risk user → budget priority; high-debt → debt management priority

Concept DAG — 10 Financial Topics

DFS cycle detection validates acyclic prerequisite chain:

  • income_basics → expense_tracking → budgeting_basics
  • budgeting_basics → emergency_fund → debt_management

Software Engineering
Design Patterns and Deployment

Key SE Design Patterns Applied

  • Multi-Agent Architecture — three autonomous agents with strict separation of concerns
  • Two-Tier Pipeline — Traditional ML (Tier 1) + Agentic LLM (Tier 2) with fallback
  • Singleton — GroqClient, ConceptGraph, FinBERT pipeline, and Supabase client
  • Strategy Pattern — Dual-DB routing (SQLite vs Supabase) with env-var control
  • Observer Pattern — Android NotificationListenerService monitors notification stream
  • DAG Structure — Concept prerequisite graph with DFS cycle detection
  • Bayesian Inference — probabilistic knowledge state; not deterministic scores
  • RAG Pattern — TF-IDF retrieval + tag-bonus weighting grounds all LLM responses
  • CORS middleware · Auth guards · Background startup tasks · Response caching

Cloud Deployment Stack

  • Docker: python:3.11-slim base; PyTorch CPU + spaCy + transformers in multi-stage build
  • Render.com: render.yaml declarative auto-deploy with env variable injection
  • HuggingFace Spaces: alternative target on port 7860 via git subtree push
  • Startup optimisation: FinBERT (500 MB) pre-loaded in daemon thread + NSE cache pre-warmed for zero cold-start latency

Required Environment Variables

  • SUPABASE_URL and SUPABASE_SERVICE_KEY
  • GROQ_API_KEY
  • FIREBASE_ADMIN_SDK_JSON (base64-encoded)

Dual-DB routing: USE_SUPABASE = bool(os.getenv("SUPABASE_URL")) — seamless dev ↔ prod switch with zero code changes.

Technical Challenges

  • Noisy SMS diversity: bank message formats vary widely — required 3-tier extraction cascade (regex → keyword-proximity → digit fallback)
  • FinBERT label mapping: id2label {0: positive, 1: negative, 2: neutral} required explicit verification to prevent silent sentiment inversion
  • Early-month low-risk bias: sparse data on days 2–5 caused underestimated risk — fixed with full-month proration projection logic
  • Boilerplate pollution: 13 regex patterns required to strip editorial noise before FinBERT inference
  • Bayesian normalisation: explicit renormalise step required after every BKT update to maintain sum = 1.0

Key Architecture Decisions

  • LLM fallback everywhere: every Groq feature has a deterministic fallback — 100% core feature availability
  • Backend as parsing authority: Android sends all notifications without client-side filtering — cleaner, more accurate classification
  • Dual-DB env switch: seamless SQLite to Supabase transition via single environment variable
  • Background FinBERT preload: 500 MB model loaded at startup in daemon thread — eliminates first-request cold-start latency
  • Card cache invalidation: mastery level change triggers fresh AI card at the appropriate difficulty
